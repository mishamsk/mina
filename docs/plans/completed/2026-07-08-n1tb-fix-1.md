# Plan: n1tb fix 1 — restore typed accounts ActiveUsage under the batched deleteability query

Undo a type-safety regression introduced during the jrdt review loop: accounts lost their typed per-source `ActiveUsage` struct in favor of an untyped id-set (`map[int64]struct{}`), while categories/tags/members kept typed usage in the same file. Rebuild the batched query on the typed foundation — same single-query efficiency, no behavior change.

## Plan Context

- Do not run review-loop.
- Backend-only. No HTTP/OpenAPI changes (the `Account.deletable` flag and all endpoint behavior stay byte-identical); no frontend changes.
- Current state (as of commit `c26dc2b8`):
  - `internal/store/dictionary_usage.go:12-73` — `AccountStore.ActiveDependentAccountIDs(ctx, ids) (map[int64]struct{}, error)`: one UNION query over journal records / transaction-template records / credit-limit history returning a bare id-set; the per-source information is discarded at the SQL level (`UNION` of bare ids).
  - `internal/services/accounts/accounts.go` — repo interface entry `:124`; `Delete` guard consumes the set (`:488`); listing deleteability consumes it (`:517`). No typed `ActiveUsage` exists for accounts.
  - Contrast: `CategoryStore.ActiveUsage` (`dictionary_usage.go:75+`) and the tags/members equivalents return typed per-source structs (`categories.ActiveUsage` etc.) with `HasActiveDependents()`.
- Required shape (mirror the sibling dictionaries):
  - Restore `accounts.ActiveUsage` in `internal/services/accounts` — per-source booleans (journal records, transaction-template records, credit-limit history) + `HasActiveDependents()`, documented like the categories one.
  - Replace the store method with a batched TYPED variant (keep it one round-trip — e.g. the same UNION with a source discriminator column, or three IN-list EXISTS passes in one query — scanning into `map[int64]accounts.ActiveUsage`; absent id ⇒ zero-value usage ⇒ deletable). No per-row queries.
  - Service: `Delete` guard and the listing's deleteability both derive from the typed usage (`!usage.HasActiveDependents()`), keeping the single shared predicate so the flag can never disagree with delete behavior.
  - Update the repo interface, store unit tests (assert per-source flags for an account with each dependent kind and one with none), and any service tests touching the old method. Integration tests should not need changes (JSON unchanged) — keep them green.
  - Update `internal/services/accounts/PACKAGE.md` only if it documents the usage contract.
- Protect — do not regress: `Account.deletable` semantics and all listing/delete behavior (integration tests must pass unchanged); the n1tb removals stay removed (no delete-by-path resurrection); single-round-trip batching (no N+1).
- Scope exclusions: no categories/tags changes (parity work is kata `cdd0`, which will mirror this corrected shape); no HTTP surface changes; no frontend changes.
