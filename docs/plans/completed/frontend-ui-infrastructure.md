# Frontend UI Infrastructure and Architecture

## Plan Context

- Phase 2 needs a minimal web UI foundation served by `mina serve`.
- Borrow PondPilot concepts from `~/dev/pondpilot` at `ebe2239`: `components`, `pages`, `hooks`, `store`, `services`, feature slices, Zustand selectors, and IndexedDB-backed UI state.
- Do not borrow PondPilot DuckDB WASM, PWA, Cloudflare, Yarn, or browser-database ownership.
- REST API paths stay at the root. Serve the web UI under `/ui/`.
- The frontend must use generated REST client code from `api/openapi.yaml`.
- Current npm registry snapshot for planned pins: React `19.2.7`, pnpm `11.9.0`, Vite `8.1.0`, Playwright `1.61.1`, ESLint `10.6.0`, Prettier `3.9.0`, Zustand `5.0.14`, idb `8.0.3`, `@hey-api/openapi-ts` `0.99.0`.
- TypeScript tooling is split: TS 6 is exposed as `typescript` for ESLint and `typescript-eslint`; TS 7 is used only for compiler checks.
- Use package aliases like `typescript: npm:@typescript/typescript6@^6.0.0` and `typescript-7: npm:typescript@rc`; lockfile resolution pins the exact installed builds.
- Resolve and pin exact non-TypeScript versions again at implementation start if the registry has moved.
