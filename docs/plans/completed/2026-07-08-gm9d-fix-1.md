# Plan: gm9d fix 1 — stale deleteability after mutations, outside-click overlay ownership, Esc-order coverage

Fix review findings from the gm9d operator review: quick toggles and side-panel saves wrongly disable the row Delete action by clobbering `deletable`; clicks inside overlays not "owned" by a panel wrongly close it; the Esc-ordering acceptance lost its e2e coverage; plus two small cleanups in the new code.

## Plan Context

- Do not run review-loop.
- Frontend-only. No ground-truth doc edits, no API changes.
- Defects (live-verified and code-traced, line numbers as of commit `b634006`):
  1. **Stale `deletable` after mutations (major).** `Account.deletable` is populated only in `listAccounts` responses; create/update responses omit it. `refreshAccountsAfterMutation` (`frontend/src/features/accounts/use-accounts-resource.ts:122-139`) refetches the fresh snapshot and then `mergeAccountsPageAccount` (`frontend/src/store/accounts.ts:344-373`) overwrites the fresh account with the flag-less mutation response, so `accounts-tree.tsx:746` (`account.deletable !== true`) disables Delete with the false tooltip "Account has active dependent records." Live repro: on `/accounts`, the demo `trading:EUR` row's Delete is enabled; toggle its featured star; Delete is now disabled until a full reload. Same happens after hide toggles and side-panel saves.
  2. **Outside-click ownership too strict (minor).** `frontend/src/hooks/use-outside-pointer-close.ts:17-42` exempts portal overlays (popovers, tooltips, `aria-modal` dialogs, page-help) only when the panel "owns" them via `contains`/`aria-controls`/`aria-describedby`. A pointer-down inside an overlay owned by another surface (e.g. a keyboard-opened toolbar filter popover while the detail panel is open, or a credit-limit dialog on the account page while the peek panel is open) counts as outside and closes the panel. Per `docs/webui-design.md` Overlays, only clicks on the underlying content perform-and-close; clicks inside any floating overlay content must never close the panel.
  3. **Esc-ordering coverage dropped (minor).** The e2e "Escape closes filter popover before transaction detail panel" was rewritten into an outside-click test (`frontend/tests/e2e/transactions-page.spec.ts:1999`), leaving the surviving Esc-ordering code path (`transaction-detail-panel.tsx:489-499`) untested. It is still reachable: open the detail panel, then open the Add-filter popover with the keyboard (no pointer-down outside), then press Esc — popover closes first, panel stays; second Esc closes the panel.
  4. **Cleanups (nits).** Dead toggle branch in the overflow select: `row-actions.tsx:72-83` handles `onToggle` for overflow items but `overflowActions` filters to button-class only (`:92`). And `use-outside-pointer-close.ts:6` hardcodes the Mina-specific `[data-page-help-content]` selector inside a generic hook (`frontend/src/hooks` is for hooks reusable outside Mina per `docs/frontend-architecture.md`) — accept extra exempt selectors via an option instead, with the Mina selector supplied by the callers.
- Protect — do not regress: everything verified in the gm9d review — outside-click close on both panels with the click still acting on the underlying content; own-dialog exemption (delete confirm keeps the panel open); focus NOT restored on outside close; hide/feature toggles and their refreshes; delete flows (leaf and group, 409 surfacing, dialog focus behavior); include-hidden toggle; fp3e affordance behavior (hover reveal, toggles visible when actions fold); collapse ladders; `just test-frontend-e2e` green.
- Scope exclusions: do not change backend deleteability semantics; do not redesign `refreshAccountsAfterMutation`'s general refresh flow; no new API calls; leave dual account/group row action composition as is.
