# Plan: e3fw review fix 1 — complete the schema-selection fix at the connection level and remove the retry masks (`e3fw`)

The branch's `USE <accounting>` fix in `internal/store/tx.go` covers only transactional paths. Non-transactional reads of the generated enum column `recurring_definition.schedule_class` (`internal/store/recurring.go:78,111,296`) still run on raw pool connections; a pooled connection that has never executed `USE` fails with `Catalog Error: Type with name recurring_schedule_class does not exist` — confirmed empirically on DuckDB in both file-backed and in-memory modes. `ListMaterializationDefinitions` runs on every `GET /api/recurring-occurrences` outside any tx, so the review page's own load path can intermittently 500 (runtime pool = 2 connections; connection #2 is uncured until its first tx). The branch shipped retry loops masking this instead of fixing it. Fix it at the root.

## Plan Context

- Do not run review-loop.
- Root fix: select the accounting schema at CONNECTION ESTABLISHMENT so every pooled connection is cured before its first statement — e.g., the duckdb-go connector's connection-init hook (boot/init queries) or an equivalent single ownership point where connections are created (`internal/store` owns the DB open path). After that, the per-tx `USE` in `tx.go:24,42` becomes redundant — prefer exactly one mechanism: move the `USE` to connection init and revert the per-tx calls (keep `migrations.go`'s signature change only if still needed).
- Remove BOTH retry masks once the root fix lands:

  - `frontend/tests/e2e/recurring-page.spec.ts:296-303` — the Retry-button `toPass` loop; assert the load works first time.
- Restore deterministic regression coverage (commit a40d4b01 deleted the earlier attempt): a Go test that fails if the schema-selection invariant is removed — e.g., open a store with a pool of ≥2 connections and drive concurrent (or connection-exhausting) NON-transactional reads that hit `schedule_class` (ListMaterializationDefinitions / List) so the second connection must serve one; with the fix absent this must fail deterministically, with it present it passes. Place it at the appropriate boundary per `docs/TESTING.md` (store/apptest — not under internal/tools).
- Document the invariant in the same commit (CLAUDE.md obligation): a short comment at the connection-init `USE` explaining that unqualified sequence defaults and enum types resolve against the connection's current catalog/schema (mirror the tone of `internal/store/backups.go:63-66`), plus one line in `internal/store/PACKAGE.md`.
- Protect — do not regress: everything else on the branch (the §8 page, confirm/dismiss flows, Expected filter, the `recurring_definition_fqn` API extension and its apptest coverage); `just test-integration` must be run and green (REST behavior changed on this branch; the operator verified it green pre-fix — keep it green).
