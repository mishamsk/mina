# Plan: axf6 fix 1 — journal grid unusable at docked width; reconciliation UI removal; hydration and error-mapping gaps

Fix operator-review findings on the Advanced journal editor: the record grid renders as an unusably squeezed multi-column table at the docked panel width (the main defect, live-verified); the out-of-scope per-row reconciliation select; two small robustness gaps.

## Plan Context

- Do not run review-loop.
- Frontend-only. No ground-truth doc edits, no API changes.
- Defects (as of commit `9f24bde`):
  1. **Record rows unusable at the docked panel width (major, live-verified at 1440×900).** On `/transactions` with the docked entry panel (~390px wide), the Advanced grid renders header labels overlapping into gibberish ("ACCOUNTAMOUN…CURREATEGORY…MEMBERSTATUSMEMO") and each row's inputs as ~10px slivers — every field is unreadable and untypable by mouse. The committed plan required "each record laid out as a compact bordered row block (wrapping fields) consistent with the panel's existing form styling; no horizontal scrolling; keyboard reachable". Rework the row layout: labeled, wrapping fields inside a bordered per-record block (labels above or beside inputs like the shorthand tabs — not a shared table header row), readable at the panel's real width on `/transactions` at common desktop sizes (1280–1600 wide). The delete-row control follows the affordance rules. Verify LIVE with `just dev --demo` at 1440×900 (screenshot-level scrutiny: every field readable, labeled, and usable; no horizontal scrollbar), not only via e2e selectors.
  2. **Per-row reconciliation select exceeds plan scope (remove).** Review-fixer commit `9f24bde` added a reconciliation-status select per row (`entry-panel.tsx:1792-1805`). The plan excluded reconciliation UI (webui-design reserves reconciliation surfaces for Phase 5, "hidden until relevant data exists"). Remove the select; keep sending `reconciliation_status: "unreconciled"` in payloads. Keep the per-row posting-status select and date disclosures.
  3. **Draft hydration doesn't sanitize `postingStatus`** (`entry-panel.tsx:286-300` sanitizes reconciliation/tagIds/draftId but not postingStatus) — a corrupt stored value flows into the API payload. Sanitize to the valid enum with a `posted` fallback, consistent with the rest of the migrate function.
  4. **`initiated_date` API errors don't map to the date field** (`advancedFieldErrorsFromAPI`, `entry-panel.tsx:638-688`, handles only `records[N].field` and semantic-shape messages). Map messages referencing `initiated_date` onto the Advanced date field; everything else keeps its current behavior.
- Protect — do not regress: everything verified in the review — escalation fidelity for all four tabs (record shapes match the backend shorthand derivation; the escalation e2e with API-level record comparison must stay green), BigInt mantissa balance math and meter behavior, save gating, intent→account-type filtering (exact accounting-semantics table), draft persistence incl. legacy hydration, shorthand tabs and palette entry commands, `just test-frontend-e2e` green (110 tests).
- Scope exclusions: nothing else — no new fields, no meter changes, no EntityPicker behavior changes.
