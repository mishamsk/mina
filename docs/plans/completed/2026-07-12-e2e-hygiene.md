## Plan Context

- Kata issues: `eg7c` and `hspf`. One sub-branch.
- MANDATORY pre-reads: `docs/TESTING.md` (frontend e2e is a smoke suite proving wiring — deflake via real wait conditions and fixture isolation, never blanket timeouts or retries), the Playwright config (one shared demo server per project; specs share state within a run).
- eg7c items:
  1. `reference-table-layout.spec.ts` fixture "E2E Scroll Members <unique>" sorts between demo Avery/Morgan and can break `members-page.spec.ts` nth-based assertions — rename to sort after demo data (existing convention) AND prefer replacing fragile nth-order assertions with name-anchored ones where cheap.
  2. The 8-scenario (4 pages × 2 viewports) monolithic geometry test hides failures behind the first — split per page (or page×viewport) so failures localize.
  3. The frame class string + `reference-table-frame` testid is hand-duplicated across `reference-tree.tsx`, `members-page-content.tsx`, `accounts-tree.tsx` — extract one shared constant/component slot to stop drift.
- hspf: the shorthand entry session-counter spec ("entry panel creates each shorthand transaction type" area, `transactions-page.spec.ts`) flakes on chromium — root-cause the timing (proper wait-for-response/state conditions, not sleeps); acceptance: 20 consecutive solo chromium runs green.
- OPERATOR FLAKE LIST (accumulated this fleet; each failed once in a full run and passed solo/on retry — diagnose and deflake each; the common suspicion is shared-server state pollution or ordering between specs, plus missing wait conditions):
  1. `tags-page.spec.ts` "tags row actions hide groups and move renamed paths into transaction filters" — ~1/4 webkit failure rate SOLO (toContainText timing) — this one flakes even in isolation.
  2. `members-page.spec.ts:232` "members side panel creates renames and deletes members with conflict feedback" (webkit, toContainText).
  3. `transactions-page.spec.ts` "transactions page add-filter menu drives server filters and chips" (chromium, toHaveValue).
  4. `transactions-page.spec.ts` "advanced journal account picker keeps suggestions filtered but resolves exact hidden FQNs" (webkit, toBeVisible).
  5. `transactions-page.spec.ts` "transaction detail duplicate prefills a new entry" (chromium, toBeVisible).
  Also one "Accounts focus" flake was seen once (unnamed — check accounts-page.spec.ts focus assertions for missing waits).
- Techniques (in preference order): unique, sort-safe fixture naming and per-spec data scoping; assert on the specific entity rather than positional indexes; wait on the network/UI condition that the assertion depends on (waitForResponse, expect.poll on state) instead of fixed timeouts; only as a last resort reorder or isolate specs. Do NOT enable Playwright auto-retries and do NOT weaken assertions.
- Acceptance gate for the branch: `just test-frontend-e2e` green in ONE shot, three consecutive full runs (record the three run results in the kata comment), plus the hspf 20-run solo bar.
- Test-only changes expected; tiny product testid/constant extraction allowed for eg7c(3). No ground-truth docs, no PROJECT_STATE.md.
