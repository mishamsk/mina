Everything else on the branch was verified sound — protect it.

## Plan Context

- Do not run review-loop.
- Findings (operator audit, file:line):
  1. HIGH — `frontend/src/features/ledger/entry-panel.tsx:1941-1944`: `updateActiveTab` routes every Advanced-tab click through `editActiveTabAsJournal()` whenever a shorthand tab is active, including plain CREATE mode. A user with in-progress advanced journal records who visits a shorthand tab and clicks Advanced again gets their advanced draft silently overwritten by a conversion of the shorthand form. Escalation-on-tab-click is an EDIT-mode behavior: gate that branch on the active `replacement` launch; in create mode the Advanced tab must remain a pure switch to the independently persisted advanced draft (the per-tab draft contract in the ledger PACKAGE.md).
  2. MEDIUM — `entry-panel.tsx:1580`: the launch-init effect depends on the `lookups` snapshot, whose identity changes on every `refreshLedgerLookups` (e.g. command-palette entity creation); a mid-edit refresh re-runs init and silently resets unsaved edit-session values to the transaction's originals. Make launch initialization run once per launch (e.g. keyed by a launch identity/ref), not per lookups identity, while still reading current lookups when it does run.
  3. MEDIUM — `frontend/src/features/command-palette/command-palette.tsx:453` + the transactions page `entryLaunch` lifecycle: palette "New transaction/spend/…" commands call the store `openTransactionEntryPanel(entryType)` without clearing the page's active `entryLaunch`, so the remounted panel re-enters edit/split mode and ignores the requested tab. Clear/supersede the launch when a plain new-entry open happens (own the fix at the appropriate layer — page lifecycle or store signal — without coupling the palette to page internals more than the existing pattern does).
- Add regression coverage:
  - e2e: per-tab draft independence in create mode — enter records in the Advanced tab, switch to Spend, click Advanced again: the advanced records must still be there (this must FAIL on the unfixed code); plus one discard-dialog path (launch Edit over a non-empty persisted draft → "Discard draft"/"Keep draft" dialog appears; Keep preserves).
  - e2e or targeted coverage for the palette flow: with an edit launch active, invoking a palette new-spend command opens a clean create-mode spend tab.
- Protect — do not regress: everything verified sound — fit detection, verbatim replacement passthrough (`amount_usd`, timestamp fraction), duplicate-as-create, split-through-journal, same-id e2e assertions, draft persistence for create flows, pickers' hidden-entity inclusion, detail-panel actions, all existing suites.
- Out of scope (noted for kata follow-up, do not implement): expected-occurrence edit dead-end; nanosecond timestamp blanking the datetime-local control; the minor non-additive e2e tweaks.
- Kata `ksw0` is closed; append the fix evidence, do not reopen.
