# Plan: Generate the shared REST operation catalog and typed invokers

Extend `internal/tools/surfacegen` to emit deterministic generated code inside `internal/httpclient`: resolved per-surface metadata, input descriptors, and typed per-operation invokers calling the generated `ClientWithResponsesInterface`, with generation and freshness checks integrated into the existing Justfile/pre-commit recipes. No CLI or MCP surface consumes the catalog yet; current REST app behavior stays green.

## Plan Context

- Owning docs: `docs/cli-mcp-architecture.md` (Generation, Package Boundaries, Checks and Verification), `docs/architecture.md` (`internal/httpclient` owns "generated operation catalog and invokers"; generation is build-time tooling under `internal/tools`), `internal/httpclient/PACKAGE.md`, `docs/TESTING.md` (no unit tests in `internal/httpclient`, no tests under `internal/tools`), Justfile `openapi`/`openapi-check`/`surface-check` recipes, `.pre-commit-config.yaml` `mina-openapi-check` hook.
- Inputs: `api/openapi.yaml` (kin-openapi, already loaded by surfacegen) plus `api/client-surfaces.yaml`. Validation from the existing `-check` mode remains the gate; generation must refuse to emit when validation fails.
- Generated output: one or more `*.gen.go` files in `internal/httpclient` (for example `surfaces.gen.go`) with a `// Code generated by surfacegen. DO NOT EDIT.` header, gofmt-formatted, and byte-deterministic across runs (sorted by operationId; no map iteration order, timestamps, or absolute paths in output).
- Hand-written supporting types live in a new hand-written `internal/httpclient` file (for example `catalog.go`), documented per godoc conventions:
  - operation metadata: operationId, HTTP method, path template, summary/description from OpenAPI, per-surface presence with resolved CLI area+command name, resolved MCP group+tool name plus the four annotation booleans;
  - input descriptors rich enough for later generic CLI and MCP registrars: ordered path parameters (name, type, description, enum values if any), query parameters (name, type, required, array-ness with item type, enum values, description), and a body descriptor (present/absent, required, resolved top-level object schema info: property names, types, required list, enum values, array item types, whether the schema fits the documented simple-body rule after `$ref` resolution — one top-level object, no composition, no free-form additional properties, all properties non-null scalar/enum/array-of-those);
  - one transport-neutral result type normalizing HTTP status, response headers, and raw response body bytes — no re-parsing, no duplicated domain or transport validation;
  - one uniform invoker signature taking `context.Context`, the `*ClientWithResponses` (or `ClientWithResponsesInterface`), and a transport-neutral input (path values in template order, query values, optional raw JSON body bytes).
- Generated invokers: one per operation that is exposed on at least one surface. Operations excluded from both surfaces get no invoker and no catalog entry. Each invoker:
  - converts the transport-neutral input into the operation's generated typed params (path parameter and query parameter conversions emitted per the OpenAPI schema; conversion failures return input errors without calling the server — they must not duplicate domain validation, only the string/typed mapping the generated client requires);
  - for body operations calls the generated `<Operation>WithBodyWithResponse(ctx, "application/json", bytes.NewReader(body), ...)` variant so JSON nulls and unknown fields reach REST validation unchanged; never marshals through typed request structs;
  - for non-body operations calls the matching `<Operation>WithResponse` method;
  - normalizes the returned `Body []byte`, `HTTPResponse.StatusCode`, and `HTTPResponse.Header` into the shared result type.
- Catalog registration sets: the generated file must expose the exact per-surface sets (for example `CLIOperations()` and `MCPOperations()` or equivalent) so later registrars can enumerate them; each set must equal the configured exposures for its surface by construction. Compilation of the generated invokers against `ClientWithResponsesInterface` is the proof that configured operations still match generated REST client methods.
- Justfile integration:
  - extend the `openapi` recipe (or add a `surface-gen` recipe the `openapi` recipe calls) so regeneration is one command;
  - extend `surface-check` (or `openapi-check`, following the existing tmpdir+cmp pattern) to fail when the committed generated output differs from fresh generation — stale, missing, or hand-edited output must fail `just pre-commit`;
  - extend the `mina-openapi-check` hook `files` pattern to cover the new generated file(s).
- Freshness/determinism smoke (manual, no committed violations): (a) hand-edit one generated line, run the freshness check, observe failure, restore; (b) delete the generated file, observe failure, regenerate; (c) run generation twice and `cmp` outputs byte-identical; (d) temporarily flip one operation's MCP decision from excluded to exposed (`getHealth`) with valid names/annotations, regenerate, confirm the invoker and MCP set entry appear, then revert config and regenerate. Report observed results.
- Update `internal/httpclient/PACKAGE.md` (catalog/invoker ownership, generated-file freshness note alongside the existing openapi note) in the same commit that lands the generated output.
- No tests anywhere: no unit tests in `internal/httpclient`, no tests under `internal/tools`, no new app-test scenarios (existing app-tests plus compilation and the smokes above are the evidence).
- Do not modify `internal/clientcli`, `internal/mcpserver`, `internal/runtime`, `cmd/mina`, or REST server behavior. Do not edit `internal/httpclient/openapi.gen.go`.
