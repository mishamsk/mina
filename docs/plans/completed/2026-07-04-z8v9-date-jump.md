# Plan: Jump-to-date control on the Transactions page — Kata issue `z8v9`

Add the date-jump control from the Transactions screen spec (`docs/webui-design.md` §2: "A date-jump control navigates to any point in history"), wired to the transactions list API's date-anchored pagination (`anchor_date`, effective `offset` in the response) and synchronized with the pager and URL state.

## Plan Context

- Ground truth: `docs/webui-design.md` (§2 Transactions; Tables rules; URL state rules), `docs/frontend-architecture.md` (REST data access, page snapshots keyed by normalized request params, URL as shareable state), `api/openapi.yaml` (`listTransactions`), `docs/webui-theme-arcade-cabinet.md` (control styling). Read before starting.
- API contract (already shipped, no backend changes): `GET /api/transactions` takes `anchor_date` (`api/openapi.yaml:1205-1212`) — returns the page containing the first transaction at or before that initiated date, clamps to the oldest page, valid only with `initiated_date` desc, overrides `offset`. The backend snaps the effective offset to a page boundary given `limit` (`internal/store/transactions.go:378-412`) and the response's `offset` field is the effective offset (`TransactionListResponse`, `openapi.yaml:3426-3429`). Generated types already expose `anchor_date` (`frontend/src/api/generated/types.gen.ts:2041`) and the response `offset` (`:607-617`).
- Current frontend gaps (line numbers as of this plan's commit):
  - `TransactionPageParams` is `{limit, offset}` only; `fetchTransactionPage` hardcodes sort and does not pass `anchor_date` (`frontend/src/api/ledger.ts:21-37`).
  - Snapshot key is `${limit}:${offset}` and the response's effective `offset` is discarded (`frontend/src/store/transactions.ts:56-57`, `frontend/src/features/ledger/use-transactions-resource.ts:62-66`).
  - `PageHeader` has an unused `toolbar` slot (`frontend/src/features/app-shell/page-header.tsx:11,41-48`) — the mount point.
  - No date-picker component exists; the entry panel uses a themed native `<input type="date">` (`frontend/src/features/ledger/entry-panel.tsx:780-792`); date helpers in `frontend/src/utils/date.ts`.
  - Page URL writers `setPage`/page-size use the object `setSearchParams({page, pageSize})` form, which clobbers other params like `?transaction` (`frontend/src/pages/transactions-page.tsx:123-128,174-179`); the detail hook uses the functional form to preserve params (`use-transaction-detail.ts:100-115`).
- Operator decisions (do not relitigate):
  - The jump is an action, not persistent filter state: no `?anchor` URL param. Picking a date issues an anchored request; when the response arrives, the page URL updates to the effective page (`offset/limit + 1`) via the existing `?page` param. Back/forward and sharing work through `?page` as today.
  - No duplicate fetch: an anchored response is stored in the snapshot under the key of the page it landed on (`${limit}:${effectiveOffset}`) so the subsequent render of that page hits the snapshot. In-flight dedup for the anchored request itself may use a transient key including the anchor.
  - Control UX: in the `PageHeader` toolbar slot — a labeled themed native date input (reuse the entry panel's input treatment), applying the jump on change/Enter; while the anchored request is in flight the current rows stay visible (no skeleton flash, per Tables rules). Jumping again with the pager (next/prev) simply continues from the landed page. No entry-panel coupling (the design does not tie the entry date picker to the jump control).
  - While touching the page's URL writers, switch `setPage`/page-size writers to the functional `setSearchParams` form so they stop clobbering `?transaction` (they funnel the date jump too).
- Preserve, do not regress: server pagination e2e ("transactions page uses server pagination controls"), page snapshot render-from-memory behavior, rows-stay-visible-while-loading behavior, detail deep links.
- This delivers a `docs/webui-design.md` Phase 2 Transactions feature: include the PROJECT_STATE.md update in the final commit. No package-doc changes expected (no cross-package contract changes); update `frontend/src/store/PACKAGE.md` or feature docs only if snapshot-keying semantics described there change.
